# Gotchas

Things that will cost real debugging time if you don't know them up front.

## Server-import leak

**The single most important rule.** TanStack Start route components run on both
server and client. If `@/db`, `postgres`, `drizzle-orm`, or `perf_hooks` is
referenced at the **module scope** of anything reachable from a client bundle,
the DB driver gets pulled into the browser build and **hydration breaks**
(classically: `Buffer is not defined`).

Keep those imports **inside server-fn / `createServerOnlyFn` handler bodies**, in
`src/server/*` or `src/db/*`. Client/route files import the *server function*, not
the db. After touching this boundary, verify the client bundle is clean:

```bash
pnpm build
grep -rl "drizzle-orm\|perf_hooks\|from \"postgres\"" .output/public/   # expect NO matches
```

`src/lib/*` is mostly client-safe; the exceptions (`auth.ts`, `queries.ts`) only
ever get used from server contexts or pull server fns in a tree-shakeable way.
Don't add a top-level `db` import to a `lib` file.

## `useLiveQuery` is client-only

It calls `startSyncImmediate()` during render with no server snapshot, so it
throws / fetches a relative URL under SSR. Always gate it behind a mount flag
(`useMounted()` in `src/routes/_authed/shopping.tsx`). See
[realtime-shopping-list.md](./realtime-shopping-list.md).

## Electric shape URL must be absolute + proxy must pin columns

Relative shape URLs silently fail; `columns` must include the collection's key
column. Covered in [realtime-shopping-list.md](./realtime-shopping-list.md#gotchas-specific-to-realtime).

## Dev-only recipe-image passthrough shim

Vite's dev server routes any request with `Sec-Fetch-Dest: image` (every `<img>`)
to its static-asset pipeline, which **404s** the dynamic `/api/recipes/:id/image`
server route. `vite.config.ts` has a `serve`-only middleware that strips that
header so the request reaches the TanStack route. Production (Nitro) doesn't need
it. If recipe images 404 in dev, this shim is why — don't remove it.

## Service worker (PWA) can serve stale assets

`public/sw.js` is hand-rolled (vite-plugin-pwa broke under Nitro). It only
registers in production/HTTPS and **never caches `/api`**. In dev, `__root.tsx`
actively **unregisters** any SW and clears caches — because a SW left over from a
`vite preview`/prod build on the same `localhost` origin will keep serving stale
HTML/assets that survive hard refreshes (and breaks things like newly added
routes). If dev behaves impossibly staleness-wise, suspect a leftover SW.

## Env vars are read at boot

`DATABASE_URL`, `BETTER_AUTH_*`, and `ELECTRIC_*` are read at server start.
Changing `.env` requires a **dev-server restart**. `.env` is gitignored — never
commit secrets.

## Dev-server port sprawl

Dev runs on **port 3001** (`pnpm dev`). It's easy to end up with several stray
dev servers on incrementing ports (3002, 3003, …) if one is already bound — reuse
3001 and kill the others rather than letting them pile up. Cookies/sessions are
**per-origin (per-port)**, so a session created on `:3001` won't exist on a
different port.

## strict TS / unused imports

`tsconfig` has `noUnusedLocals`, `noUnusedParameters`, and `verbatimModuleSyntax`.
Leftover imports or value-importing a type will fail `tsc`. Use `import type`, and
run `pnpm typecheck` before committing.

## `routeTree.gen.ts` is generated

Generated by the TanStack Router plugin (`tsr generate`). Don't edit it by hand.
If a newly added route file isn't recognized, run `pnpm generate-routes`.

## Historical column names

`recipe.created_by` is the **owner** id (kept from an earlier shared-collection
model). `shopping_check.user_id` and the `scope_id` columns hold a **household
id**, not a user id. Don't infer semantics from these names — see
[data-model.md](./data-model.md).
